## 1. ブランドの核

### 表記

- 日本語名: 風音
- 英語名: Kazane
- カテゴリー名: Chronicle Work OS
- 基本表記: **風音（Kazane）— Chronicle Work OS**

### タグライン

> 仕事の気配を聴き、判断の来歴を綴る。

### 一文説明

> 風音は、人間とAIエージェントの仕事を、文脈、引き継ぎ、証跡、レビュー、責任境界とともに管理する、ローカルファーストの業務OSです。

### 30秒説明

AIに仕事を任せると、作業は速くなります。一方で、「なぜ始めたのか」「何を根拠に判断したのか」「どこから人が責任を持つのか」が、チャットや複数のツールに散らばりがちです。

風音は、仕事を単なるタスクとしてではなく、文脈、担当、引き継ぎ、証跡、判断履歴を持つ一つの流れとして記録します。AIが進められるところはAIへ、判断が必要なところは理由を添えて人へ戻し、意味と責任を失わない協働を支えます。

## 2. 想定する利用者

### 主な対象

- 実務でAIエージェントを使い始めた小規模組織
- AIへ仕事を任せたいが、判断根拠や責任の所在を曖昧にしたくない経営者・業務責任者
- 複数の人間とAIの間で、仕事の引き継ぎやレビューを行うチーム
- 機微な業務文脈を、まずローカル環境で管理したい利用者

### 起きている問題

- AIとの会話は残っていても、仕事の目的や結論までの道筋を追えない
- AIが「完了」と報告しても、何を確認したのか分からない
- 人間に判断を戻すべき場面が定義されていない
- 引き継ぎのたびに、背景説明を最初からやり直している
- タスク、資料、会話、Issue、判断が別々の場所に散らばっている
- 元の依頼や意図から、成果物の意味がずれても気づきにくい

### 3.2 課題提起

#### 見出し

> AIが働くほど、「なぜ」と「誰が」が見えにくくなる。

#### 本文

AIは、調査、下書き、整理、実行、レビューを担えるようになりました。しかし、仕事がチャットや個別ツールの中だけで進むと、目的、判断、根拠、未解決点が分断されます。

風音が扱うのは、タスクの進捗だけではありません。仕事が生まれた理由から、AIが行った作業、人間へ戻された判断、参照した証拠、次に更新すべき文脈までを、一続きの記録として扱います。

### 3.3 提供価値

#### 文脈を仕事の起点にする

依頼の背後にある問い、目的、制約、顧客・組織の事情、過去の判断を Context Card に保存します。担当が変わっても、仕事の意味を引き継げます。

#### AIの停止を正常な流れにする

価格、契約、外部公開、専門判断など、AIが越えてはいけない境界を Escalation Gate として定義します。AIは曖昧に完了せず、理由と未解決点を添えて人へ返します。

#### 成果を根拠へつなぎ直す

Evidence Log に、参照した情報源、ファイル、会話、Issue、テスト結果などを記録します。要約や成果物から、その根拠をたどれる状態をつくります。

#### 判断の変化を後から見直せる

Handoff Note と RDE Audit により、何を行い、何を判断できず、元の意図から何が変化したかを確認します。重要な判断を再検討し、必要なら戻せる来歴を残します。

### 3.4 仕事の流れ

#### 見出し

> 仕事の発生から、判断と改善までを一つの流れに。

1. **文脈を記録する** — Context Card に、問い、目的、制約、過去の判断を残します。
2. **仕事をつくる** — Work Item を作成し、人間またはAIへ割り当てます。
3. **AIまたは人が進める** — 作業状況と担当を共通の Work Board で確認します。
4. **引き継ぐ、または止める** — Handoff Note に実施内容、不確実性、次のアクションを記録します。責任境界に達した場合は人へ戻します。
5. **証跡と意味を確認する** — Evidence Log と RDE Audit で、根拠と意味の変化を検査します。
6. **次の文脈へ反映する** — 得られた判断や未解決点を、Context Card と次の Work Item へつなぎます。

### 3.5 主な機能

#### Flow Dashboard

進行中の仕事、AIが処理した仕事、人間の判断を待つ仕事、監査や専門家確認が必要な仕事を俯瞰します。

#### Work Board

人間とAIが同じ業務空間で Work Item を扱います。開発だけでなく、調査、資料作成、営業、顧客対応、会議、運用などの業務を対象にできます。

#### Context Cards

仕事が生まれた問い、目的、制約、過去の判断、未解決点を保存し、複数の Work Item を共通の文脈へ接続します。

#### Handoff Notes

実施内容、判断したこと、判断できなかったこと、不確実性、参照証跡、次のアクションを、AIと人間の間で引き継ぎます。

#### Escalation Gate

AIが自走できる範囲と、人間または専門家へ判断を戻す条件を定義します。停止を失敗ではなく、責任を守る正常な運用として扱います。

#### Evidence Log / RDE Audit

成果を情報源や作業結果へ結び、元の問いや意図から意味がどう変化したかを確認します。

### 3.6 利用イメージ

#### 顧客対応

問い合わせの背景を Context Card として保持し、AIが返信案を作成。契約や価格に関わる判断は人へ戻し、送信前の根拠と承認を残します。

#### 調査・資料作成

調査目的と前提を記録し、AIが参照した情報を Evidence Log へ接続。下書きが元の依頼からずれていないかを確認してから公開します。

#### 開発・運用

Work Item を Issue、変更内容、テスト結果、レビューへ接続。AIが作業した場合も、実施内容と未解決点を Handoff Note に残します。

#### AI番頭ワークフロー

メール、予定、会議、文書から生まれた仕事を一つの業務フローへ整理し、AIが準備できる部分と人間が判断する部分を分けます。

### 3.7 ローカルファーストと責任境界

#### 見出し

> 組織の文脈を、組織の手元から始める。

風音は Tauri ベースのローカルアプリとして開発されています。仕事、文脈、判断、証跡を扱うからこそ、機微な情報を最初からクラウドへ置くことを前提にしません。

また、風音は人間の責任をAIへ移すための製品ではありません。AIは作業主体になれても、法的・組織的な責任主体ではありません。重要な判断を人へ戻し、その理由と承認を記録できることを重視します。

### 3.8 現在地

#### 見出し

> 現在、v0.8の開発・自己利用段階です。

風音は現在、公開開発中の初期プロダクトです。Tauri デスクトップアプリを中心に、ローカル SQLite、Work Item、Context Card、Handoff Note、Evidence Log、Escalation Gate、RDE Audit、エージェント向け CLI / MCP 接続を実装・検証しています。

まずは開発、文書作成、調査、運用などの内部業務で自己利用し、AIと人間の協働フローを改善しています。一般向けの安定版ではないため、機能、データ形式、導入方法は変更される可能性があります。

### 3.9 風音という名前

> 風音は、目に見えない変化を知らせる音です。

仕事にも、まだ明文化されていない気配があります。小さな違和感、判断を待つ沈黙、引き継がれなかった背景、変わり始めた前提。風音は、そうした兆しを拾い、後からたどれる記録へ変えていくことを目指しています。

## 4. FAQ原稿

### 風音はタスク管理ツールですか？

Work Item とボードを備えていますが、一般的なタスク管理だけを目的としていません。仕事の背景となる文脈、AIと人間の引き継ぎ、参照した証跡、責任境界、判断の変化までを一つの流れとして扱います。

### 開発チーム向けの製品ですか？

開発は最初の検証対象の一つですが、開発専用ではありません。調査、資料作成、営業、顧客対応、会議、運用など、AIが関与できる業務フロー全般を対象にしています。

### AIが自動で仕事を完了してくれますか？

AIが実行できる仕事は任せられますが、完全自律を目的としていません。重要な判断や権限の境界に達したとき、AIが理由と未解決点を残して人へ戻せることを重視しています。

### データはどこに保存されますか？

現在のデスクトップ版は、ローカル SQLite を中心に開発しています。将来の外部接続やリモート利用は、明示的な認証、権限、監査、データ境界を伴う構成として検討します。

### 今すぐ利用できますか？

現在は v0.8 の開発・自己利用段階で、一般向けの安定版ではありません。ソースコードと開発状況は GitHub で公開し、導入可能な範囲と制約を明示しながら開発を進めています。

### RDE Auditとは何ですか？

成果物の品質だけでなく、元の問い、文脈、判断意図から意味がどのように保存・変換・補完されたかを確認するための監査です。AIによる要約や下書きが、当初の目的から静かにずれる問題を見つけやすくします。

## 7. 制作時に必要な素材

優先度の高い順に用意します。

1. Flow Dashboard の実画面キャプチャ
2. Work Item から Context、Handoff、Evidence へ移る流れが分かる3枚程度の画面
3. Escalation Gate でAIが人へ判断を戻す場面
4. アプリアイコン、ロゴ、ダーク背景用ロゴ
5. GitHub リポジトリURL
6. 対応環境と導入条件を記載した開発版案内
7. 更新情報を掲載する場所

画面キャプチャには、実在する顧客名、メールアドレス、秘密情報、ローカルファイルパスを含めません。デモデータであっても、公開用であることが分かる内容へ統一します。

## 8. 原稿運用ルール

- 「現在できること」と「将来目指すこと」を同じ時制で書かない。
- 機能名だけでなく、利用者にとって何が変わるかを併記する。
- 「AIが止まる」ことを失敗ではなく、責任境界を守る正常な流れとして説明する。
- 「ローカルファースト」は「クラウドを使わない」という意味にしない。
- 「業務OS」は完全自律や既存OSの置き換えではなく、仕事・文脈・来歴を接続する基盤として説明する。
- 開発方法論や内部用語は、トップページの理解に必要な範囲に限定する。
- バージョン、対応OS、配布方法、導入可否は、リリースのたびに実装と照合する。
