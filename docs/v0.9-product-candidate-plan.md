# v0.9 Product Candidate Plan

`v0.9 utsuwa / 器` turns the verified local prototype into an operable local
product candidate. It does not expand the product into a cloud service.

## Release boundary

In scope:

- recoverable local data;
- explicit human and agent roles with enforced permissions;
- first-run and support operations that do not require repository knowledge;
- hardened provenance replay;
- a reproducible signed macOS candidate build;
- honest trust, support, pricing, and product materials.

Out of scope:

- general public availability or a stable `v1.0` promise;
- cloud-first storage or synchronization;
- production Remote GUI, external IdP selection, billing, or multi-tenant service;
- claiming v0.8 human validation before an observed external trial passes.

Remote GUI remains a design gate in v0.9. ADR-0014 must define read-only and
approval-only roles, session scope, relay topology, and audit behavior before a
separate implementation milestone is approved.

## Exit criteria

v0.9 is complete only when:

1. a real external trial has a recorded result and unresolved findings are linked;
2. backup and restore preserve Work Items, Context, Handoffs, Events, Evidence,
   Gate Rules, Agent Profiles, identity, and audit records;
3. an unauthorized role is denied in both automated tests and a recorded runtime
   scenario, while an authorized action remains auditable;
4. a clean macOS user can install, start, understand the data location, create the
   first workflow, export diagnostics, and recover from backup using documentation;
5. Replay remains inspectable with missing records and a long history;
6. product, pricing, trust, privacy, and support materials state current behavior,
   non-promises, and responsibility boundaries;
7. the candidate build and rollback procedure are reproducible from a tagged commit.
